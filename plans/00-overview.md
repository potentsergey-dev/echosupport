# EchoSupport — обзор проекта

## Видение

**EchoSupport** — это SaaS-платформа для встраивания AI-ассистента клиентской поддержки на любой сайт. Владелец сайта через панель администратора:

1. Создаёт «агента» (имя, аватар, роль, system-prompt).
2. Загружает базу знаний (PDF/TXT/MD-файлы) и/или указывает URL для краулинга.
3. Запускает индексацию (контент → эмбеддинги → векторная БД).
4. Получает один JS-сниппет, который вставляет на свой сайт.

На сайте клиента появляется виджет чата, который:

- Принимает текстовые и голосовые вопросы (микрофон → STT).
- Отвечает на языке вопроса (RAG поверх загруженных знаний + LLM).
- Помнит контекст текущей сессии и автоматически чистит данные через TTL.

## Ключевые требования (от пользователя)

| #   | Требование                                                | Решение в проекте                                                                                            |
| --- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 1   | Развёртывание на hoster.by «Оптима»                       | Node.js приложение через Passenger/PM2 (тариф поддерживает Node.js)                                          |
| 2   | Multi-tenant (несколько сайтов в будущем)                 | Архитектура изначально multi-tenant, для MVP — один тенант                                                   |
| 3   | Панель настройки агента (повторно открываемая)            | SPA-админка `/admin` с авторизацией                                                                          |
| 4   | Имя, аватар, инструкции, файлы, URL, приоритет источников | Поля в форме агента + флаг `source_priority` (`files` / `url` / `merge`)                                     |
| 5   | Кнопка «Проиндексировать» и «Сохранить»                   | Отдельные действия: сохранение настроек ≠ запуск индексации                                                  |
| 6   | Виджет с текстом и микрофоном                             | Preact-виджет + MediaRecorder API + Deepgram STT                                                             |
| 7   | Индикатор «{Имя агента} печатает…»                        | Streaming через SSE: индикатор пока не пришёл первый токен, далее — постепенный вывод                        |
| 8   | Контекст разговора в рамках сессии                        | Last-N сообщений + summary при превышении окна                                                               |
| 9   | Авто-удаление диалогов                                    | TTL `expires_at` в БД + фоновая задача очистки                                                               |
| 10  | Авто-определение языка                                    | Современные LLM (GPT-4o-mini / Claude Haiku) определяют язык сами; STT Deepgram поддерживает мультиязычность |

## Принципы проектирования

- **Cost-first**: pay-per-use везде, где можно (OpenRouter, Deepgram, Qdrant Cloud free tier, Neon free tier).
- **Vendor-agnostic**: всё, что общается со сторонним API, скрыто за интерфейсом-адаптером, чтобы можно было заменить провайдера.
- **Multi-tenant ready**: схема БД с `tenant_id` с первого дня, даже если используется один тенант.
- **Privacy by default**: PII не логируется, секреты шифруются (AES-256-GCM), TTL на диалоги.
- **Embeddable**: виджет — самодостаточный JS-bundle, ничего не ломает на сайте клиента (Shadow DOM).
- **Realtime UX**: streaming ответов через SSE — пользователь видит ответ в процессе генерации.

## Что НЕ входит в MVP (Phase 0–10)

- Биллинг и тарифы — Phase 11.
- Регистрация публичных пользователей (только один админ-аккаунт владельца) — Phase 11.
- Аналитика/дашборды (только базовые счётчики) — Phase 11.
- Голосовой ответ (TTS) — Phase 14+.
- Эскалация на живого оператора — **Phase 10.5** (см. [`plans/09-phase-10.5-operator-inbox-and-booking.md`](09-phase-10.5-operator-inbox-and-booking.md)).
- Запись на приём (booking) — **Phase 10.6**.
- Интеграции с CRM (Bitrix24, AmoCRM) — Phase 14+.

## Дорожная карта (high-level)

См. [`plans/05-roadmap.md`](05-roadmap.md). Кратко:

- **Phase 0** — Foundation (монорепо, конфиги).
